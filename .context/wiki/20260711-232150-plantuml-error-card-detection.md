# PlantUML's onError rarely fires — invalid syntax renders as a native error card instead

PlantUML's engine follows its decades-old convention of always returning *some* SVG, even for empty, garbage, or malformed input — `onError`/promise-rejection essentially never fires for syntax errors (only for rare JS-level failures like WASM instantiation). The error is baked into the SVG as `<text>` elements, forming one of two native error-card templates (confirmed by rendering garbage/empty/broken input through a real browser):

1. **Missing/unrecognized diagram-type directive** (only reachable *before* auto-wrapping — content not yet wrapped in `@startuml`/`@enduml`): first `<text>` element reads exactly `"Diagram not supported by this release of PlantUML"` (bold header), followed by explanatory lines ("Sorry, but the following directive ... is not recognized.", "Possible causes:", "Suggested actions:", etc).
2. **Content wrapped in `@startuml`/`@enduml` but invalid inside**: ends in one red (`#FF0000`) `<text>` element matching `/\(Assumed diagram type:/` — e.g. `"Syntax Error? (Assumed diagram type: sequence)"`, `"Empty description (Assumed diagram type: sequence)"`. This card also carries an ugly, unrelated version-nag banner (`"This version of PlantUML is N days old...plantuml.com/download"`, `"Possible causes:"`, `"Suggested actions:"`) because this npm build ships with its `$version$`/`$git.commit.id$` placeholders unstamped.

Madaview's hydration (`web/src/hydrate.ts`, `extractPlantumlError`) detects these via two regexes tested first against the raw SVG string, then — if either matches — parses the SVG with `DOMParser` to find the specific `<text>` node containing the match and extract it as a clean one-line message (falling back to a generic `"invalid PlantUML syntax"` string if no single `<text>` node matches despite the whole-SVG match). The clean message replaces the SVG entirely (`block.textContent = "PlantUML render error: ..."`), hiding the version-nag banner.

**Gotcha for tests/verification**: the extracted one-line message legitimately *contains* the same substring the detection regex matches on (e.g. `"(Assumed diagram type: sequence)"` appears inside the app's own clean error text). Don't reuse the detection regex to assert "no native error card leaked" against the full page — it'll false-positive on the app's own clean message. Instead assert on markers unique to the *raw, unextracted* card: the version-nag banner strings (`"plantuml.com/download"`, `"This version of PlantUML is"`, `"Possible causes:"`, `"Suggested actions:"`), which never appear in the extracted one-liner. `e2e/render-plantuml/verify` does this.
