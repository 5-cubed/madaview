# Navigating to /settings unmounts Workspace/ContentPane, even though WorkspaceProvider's reducer state survives

`App.tsx` renders `<Routes><Route path="/view/*" element={<Workspace/>}/><Route path="/settings" element={<Settings/>}/></Routes>` — `Workspace` (and therefore every `Pane`/`ContentPane` it renders) is route-gated and **fully unmounts** when navigating to `/settings`, even though `WorkspaceProvider` itself lives above `<Routes>` and its reducer state (panes/tabs/activeTabId) survives untouched (see [[20260712-004500-popstate-vs-app-navigation-workspace-reset]] for the separate popstate-vs-in-app-nav distinction).

**Practical consequence for e2e tests and any future feature**: a `/settings` round trip preserves *structural* workspace state (pane count, tab count, which tab is active) — `e2e/tabs-and-split-view`'s checkpoint 9 confirms this — but does **not** preserve `ContentPane`'s own component-local state: `scrollTop` resets to 0, and returning to `/view/*` re-mounts fresh `ContentPane` instances, which re-fetch every currently-open tab's content via `/api/file` (one request per open tab across all panes). This is pre-existing behavior, orthogonal to and unaffected by the theme system (`.context/adr/20260712-221004-md-rendering-theme.md`) — a test asserting "zero network calls" or "scroll preserved" around a theme switch must not route through `/settings` to get there, or it will see requests/scroll-resets that have nothing to do with theming. `e2e/theme-switching/run` works around this by doing its tab/scroll/network-delta checks entirely on `/view/*` (applying themes via direct `data-theme` mutation, without any Settings navigation) and its Settings-interaction checks (hover/click, request-delta-per-click) as a separate pass that doesn't touch tab/scroll state at all.
