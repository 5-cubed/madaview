# Cut the first madaview release (v0.1.0)

## Goal
Produce a real, published GitHub Release for madaview that a user on any of the three
supported platforms (Windows, macOS, Linux) can download and run immediately, with no
build step and no runtime install (no Node, Python, or JVM) — exactly what
`.github/workflows/release.yml` and the README's Quickstart already describe, but which
has never actually been exercised: `git tag -l` is currently empty, so no tag has ever
been pushed and no release artifact has ever been published.

## Failure Criteria
- The tag push does not trigger `release.yml`, or the workflow fails at the `test`,
  `build`, or `publish` job.
- Any published binary requires installing a runtime (Node/Python/JVM) or running a
  build command before first use.
- The cross-platform smoke test (launch binary, curl `/api/status`) fails on any of the
  three runnable targets (linux/amd64, windows/amd64, darwin/arm64).
- No GitHub Release appears on the repo's Releases page after the workflow completes,
  or it's missing one of the four expected binaries (linux/amd64, windows/amd64,
  darwin/arm64, darwin/amd64).
- The release is published as "latest" instead of marked pre-release (see Direction).

## Ambiguous Zone
- `darwin/amd64` (Intel Mac) has no CI runner to execute the binary on — build-only
  verification for that one target is accepted as ambiguous-but-not-blocking, per the
  existing ADR (`.context/adr/20260711-173722-madaview-initial-setup.md`).
- OS first-run security warnings (macOS Gatekeeper, Windows SmartScreen) are expected
  and already documented in the README as a one-line user workaround — not a failure.

## Direction
Cut a real release now: push a `v0.1.0` tag to `origin`, which triggers the existing
`.github/workflows/release.yml` pipeline as-is (no changes to the build matrix — the
current four targets, linux/amd64, windows/amd64, darwin/arm64, darwin/amd64, are
sufficient). Before that tag push happens, `release.yml` needs one small change: mark
the published release as a GitHub **pre-release** (v0.1.0 is a 0.x version, not yet
declared stable) — `softprops/action-gh-release@v2` defaults to a full/"latest" release,
so the `publish` job needs an explicit `prerelease: true` (or equivalent) added. How
exactly to wire that is a /planning decision, not decided here.

Pushing the tag to `origin` is a hard-to-reverse, publicly-visible action (triggers CI,
creates a public GitHub Release under the `5-cubed/madaview` org) — it requires the
user's explicit go-ahead at execution time, not implied by this direction doc alone.

## Constraints
- No changes to the release build matrix (4 targets, as currently defined).
- No changes to the zero-build/zero-install distribution model — that architecture is
  already decided (ADR `20260711-173722-madaview-initial-setup.md`) and out of scope to
  revisit here.
- Version for this first release is `v0.1.0` (semver, pre-1.0).

## Out of Scope
- Building/publishing additional platform targets (e.g. linux/arm64, windows/arm64) —
  explicitly declined for this release.
- Reworking the release pipeline itself (trigger mechanism, test gate, smoke test
  approach) — the existing `release.yml` is trusted as-is aside from the pre-release
  flag.
- Code-signing the binaries to suppress Gatekeeper/SmartScreen warnings — not raised,
  treated as already-accepted per the README's documented workaround.
