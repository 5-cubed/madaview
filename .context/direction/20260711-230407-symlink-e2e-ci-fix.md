# Fix symlink-inside-root-followed CI failure

## Goal
The `symlink-inside-root-followed` e2e scenario passes reliably on a fresh
CI checkout (and on any fresh local clone), not just on a dev machine that
happens to have leftover state from a prior run.

## Failure Criteria
- CI continues to fail with ENOENT on the `symlink()` call because the
  scenario's `run` script assumes `data/root/` already exists as a
  directory.
- A fix that only masks the symptom for this one CI run (e.g. manually
  creating the directory once) without changing the script, so it
  regresses the next time the runner state resets.

## Ambiguous Zone
None — this is a deterministic, root-caused bug with one correct fix
location. There's no partial-success state; either the script creates its
own directory or it doesn't.

## Direction
Add `await mkdir(root, { recursive: true })` in
`e2e/symlink-inside-root-followed/run`, before the existing `rm(linkPath, {
force: true })` / `symlink(target, linkPath)` calls. This matches the
project's existing harness convention (see `e2e/root-override-ui-runtime/run`,
which does `mkdir(configHome, { recursive: true })` for its own
runtime-created directory) of scripts self-provisioning any directory they
need rather than depending on git-tracked or leftover filesystem state.

## Constraints
- `data/root/linked.md` must remain gitignored and created at runtime only
  (per the existing `.gitignore` comment: avoids symlink portability issues
  across checkouts/platforms) — this constraint is unchanged.
- Fix must live in the scenario's own `run` script, consistent with how
  other scenarios provision their runtime directories via `e2e/lib/harness.mjs`
  imports (`mkdir` from `node:fs/promises`).

## Out of Scope
- Adding a tracked placeholder file (e.g. `.gitkeep`) to `data/root/` —
  rejected as it doesn't match the project's self-provisioning convention
  and leaves the script itself still fragile.
- Any change to the symlink-following behavior/policy itself (the ADR's
  policy that symlinks inside root are followed is not in question here;
  this is purely a test-fixture setup bug).
