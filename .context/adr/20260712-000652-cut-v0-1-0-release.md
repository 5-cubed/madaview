# ADR: Cut the First Madaview Release (v0.1.0)

**Date:** 2026-07-12

## Context
Madaview's release pipeline (`.github/workflows/release.yml`) has existed since the initial
setup (ADR `20260711-173722-madaview-initial-setup.md`) but has never been exercised — `git
tag -l` is empty, no tag has ever been pushed, no GitHub Release has ever been published.
Full goal, failure criteria, and constraints live in
`.context/direction/20260711-235946-cut-v0-1-0-release.md`. This ADR covers cutting the real
`v0.1.0` release: one small workflow change (mark the release as a GitHub pre-release), then
executing the actual tag push and verifying the live result.

## Decision
- Add `prerelease: true` to the `softprops/action-gh-release@v2` step's `with:` block in the
  `publish` job of `release.yml`. No `make_latest` override needed — GitHub computes "latest"
  from the newest non-draft, non-prerelease release, so `prerelease: true` alone excludes
  `v0.1.0` from "latest" with no additional flag.
- No dry-run tag. `release.yml` already gates on the full test suite (`ci.yml` via
  `workflow_call`) before building anything, and the build/smoke-test logic is unchanged and
  already exercised on every push to main. The only new code path is the one-line prerelease
  flag, which is low-risk enough to verify directly on the real `v0.1.0` release rather than
  through a throwaway `v0.1.0-rc1` tag and its cleanup.
- No changes to `e2e/cross-platform-smoke` or any other e2e scenario — this ADR doesn't touch
  build or smoke-test logic, only the publish step's release metadata.
- No changes to the build matrix, distribution model, or pipeline trigger/test-gate/smoke-test
  design — all out of scope per the direction doc, trusted as-is from the initial-setup ADR.
- The tag push is a hard-to-reverse, publicly-visible action (triggers CI, creates a public
  GitHub Release under `5-cubed/madaview`). It is gated on the user's explicit go-ahead at
  execution time — the action sequence below pauses immediately before the push and does not
  proceed without that confirmation in the moment, regardless of this ADR's approval.
- Verification of the live run uses `gh` CLI (already authenticated as `5-cubed` in this
  environment): `gh run watch` to follow `test` → `build` → `publish` to completion, then `gh
  release view v0.1.0` to confirm the release exists, is marked prerelease, is not marked
  "latest", and has all 4 expected binaries attached.

## Design
Single-file change, no new modules or interfaces:

```yaml
# .github/workflows/release.yml, publish job
- name: Release
  uses: softprops/action-gh-release@v2
  with:
    files: dist/*
    generate_release_notes: true
    prerelease: true   # ← new line
```

Data flow for verification: `git push origin v0.1.0` → GitHub triggers `release.yml` → `test`
job runs `ci.yml` (go vet, go test, e2e Playwright suite) → `build` job matrix (4 targets, 3
smoke-tested natively per the initial-setup ADR) → `publish` job downloads all artifacts,
calls `softprops/action-gh-release@v2` with `prerelease: true` → GitHub Release created →
`gh release view v0.1.0` reads back release metadata (assets, `isPrerelease`, `isLatest`) to
confirm.

## Action Sequence
1. Edit `.github/workflows/release.yml`: add `prerelease: true` to the `publish` job's
   `softprops/action-gh-release@v2` step's `with:` block.
2. Commit that change to main (regular commit, not a tag) so the tag push in step 4 picks up
   the fixed workflow.
3. **Pause and ask the user for explicit go-ahead to push the `v0.1.0` tag to origin.** Do not
   proceed past this point without that confirmation given at execution time — approval of
   this ADR is not sufficient on its own.
4. `git tag v0.1.0` on the commit from step 2, then `git push origin v0.1.0`.
5. `gh run watch` (or equivalent polling) to follow the triggered `release.yml` run's `test` →
   `build` → `publish` jobs to completion; capture the run URL and conclusion.
6. `gh release view v0.1.0` — confirm the release exists, `isPrerelease: true`, `isLatest:
   false`, and all 4 expected binaries (`madaview-linux-amd64`,
   `madaview-windows-amd64.exe`, `madaview-darwin-arm64`, `madaview-darwin-amd64`) are
   attached as assets.
7. Update `.context/wiki/` via `/to-wiki` if new truths were found (e.g. the exact
   `prerelease`/`isLatest` behavior confirmed live) or existing entries are stale; skip
   otherwise.
8. Write a changelog entry via `/to-changelog`.
9. Remove the originating `.context/TODO.md` item via `/to-todo`, if one exists (none found at
   planning time).

## Test-Loop Design
No new e2e scenario — this ADR doesn't change build or smoke-test logic, only the publish
step's release metadata. The existing `e2e/cross-platform-smoke` scenario (reused as-is)
already covers "binary builds and answers `/api/status`" locally on every CI run; the release
workflow's per-runner smoke-test step (`release.yml`, `build` job, native OS runners) is the
same check executed on the real cross-compiled binaries for 3 of the 4 targets.

- **`run` (for this ADR, "run" = executing the real release):** reset state is implicit — no
  prior `v0.1.0` tag exists (`git tag -l` confirmed empty at planning time). Execute: push the
  `v0.1.0` tag, which triggers `release.yml`. Outputs written by GitHub Actions itself: job
  logs for `test`/`build`/`publish`, uploaded build artifacts per target, the smoke-test
  step's captured `/api/status` JSON per native runner (already part of `release.yml`, lines
  86-98), and the final GitHub Release with attached binaries. Metadata is implicit in the
  run: workflow run ID/URL, commit SHA, tag name, per-job conclusion.
- **`verify`:** `gh run view <run-id>` to check each job's conclusion is `success`; `gh
  release view v0.1.0` to check the release's asset list, `isPrerelease`, and `isLatest`
  fields against expected values. Reports good/unexpected per the Evaluation Criteria below,
  with root cause traced through the GitHub Actions job logs (available via `gh run view
  --log`) for any unexpected result.
- **Scenario:** `cut-v0.1.0-release` → tag push triggers the workflow; `test`, `build` (all 4
  matrix legs), and `publish` jobs all succeed; the 3 natively-smoke-tested targets
  (linux/amd64, windows/amd64, darwin/arm64) each return HTTP 200 with valid JSON containing
  `version` from their smoke-test step; `darwin/amd64` builds successfully (smoke test
  skipped, per the accepted Ambiguous Zone); the published release has exactly 4 binary
  assets, `isPrerelease: true`, `isLatest: false`.

## Evaluation Criteria
- **Good:** Tag push triggers `release.yml`; `test`, `build` (4/4 legs), and `publish` jobs
  all report `success` via `gh run view`; `gh release view v0.1.0` shows all 4 binaries
  attached, `isPrerelease: true`, `isLatest: false`; the 3 natively-smoke-tested targets each
  logged HTTP 200 with valid `/api/status` JSON during their build job's smoke-test step.
- **Ambiguous:** `darwin/amd64` builds successfully but has no native runner to
  execute/smoke-test it (accepted per the initial-setup ADR's Ambiguous Zone — not blocking).
  OS first-run security warnings (Gatekeeper/SmartScreen) are expected on user machines and
  already documented in the README — not evaluated here since it's a manual end-user action
  outside CI's reach.
- **Bad — maps directly to the direction doc's Failure Criteria:** the tag push doesn't
  trigger `release.yml`, or `test`/`build`/`publish` fails; any published binary needs a
  runtime install or build step before first use; the smoke test fails on any of the 3
  runnable targets; no GitHub Release appears, or it's missing any of the 4 expected binaries;
  the release is marked "latest" instead of pre-release.
