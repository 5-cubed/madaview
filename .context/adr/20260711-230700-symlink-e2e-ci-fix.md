# ADR: Fix symlink-inside-root-followed CI failure

**Date:** 2026-07-11

## Context
CI fails at `e2e: symlink-inside-root-followed` with `ENOENT` on the
`symlink()` syscall. Root cause (established in
`.context/direction/20260711-230407-symlink-e2e-ci-fix.md`): the scenario's
`data/root/` directory has no tracked contents — it holds only the
gitignored, runtime-created `linked.md` symlink. Git does not track empty
directories, so `data/root/` does not exist on a fresh checkout. The `run`
script calls `symlink(target, linkPath)` without first creating
`data/root`, so it fails wherever the directory hasn't been created by some
earlier, incidental process (true for every CI runner; false on a dev
machine with leftover state from a prior local run).

This is a test-fixture provisioning bug, not a change to the symlink
policy under test (see `.context/adr/20260711-173722-madaview-initial-setup.md`
for that policy, which is unaffected).

## Decision
Make `e2e/symlink-inside-root-followed/run` self-provision its own
`data/root` directory via `mkdir(root, { recursive: true })` before
touching `linked.md`, matching the existing convention already used by
`e2e/root-override-ui-runtime/run` (`mkdir(configHome, { recursive: true })`)
for its own runtime-created directory. Rejected alternative: a tracked
placeholder file (e.g. `.gitkeep`) in `data/root/` — fixes the immediate
symptom but doesn't match the harness's self-provisioning convention and
leaves the script itself fragile if the directory is ever removed for
another reason.

## Design
Single-file change, no new modules or contracts:

- `e2e/symlink-inside-root-followed/run`
  - Import `mkdir` alongside the existing `symlink, rm` import from
    `node:fs/promises`.
  - Call `await mkdir(root, { recursive: true })` immediately before the
    existing `await rm(linkPath, { force: true })` line, so the directory
    exists before either the cleanup or the symlink creation touches it.
  - No change to `verify`, `test`, `.gitignore`, or any other scenario
    file — they don't reference `data/root`'s existence directly, only
    `result.json` (produced after `run` already succeeds).

## Action Sequence

2. Edit `e2e/symlink-inside-root-followed/run`: add

## Test-Loop Design
No new scenario needed — this fixes the existing
`symlink-inside-root-followed` scenario's own fixture setup so it runs
correctly, rather than adding new coverage.

- **`run`:** Resets nothing pre-existing to reset (no `result/` reuse
  issue here — that's already handled by `resetResultDir`). Newly
  self-initializes `data/root/` via `mkdir(recursive: true)`, then
  (re)creates the `linked.md` symlink pointing at `data/outside/secret.md`,
  starts a server rooted at `data/root`, fetches `linked.md` through the
  API, and writes `result.json` (`{status, body}`) plus `server.log` and
  `metadata.json` (timestamp, os, arch, commit) to `result/`. This is
  unchanged except for the new `mkdir` call.
- **`verify`:** Reads `result.json` and checks `status === 200` and
  `body.includes('Secret Outside Root')`; reports `good` /
  `unexpected` accordingly. Unchanged.
- **Scenarios:**
  - `symlink-inside-root-followed` → on a checkout with no pre-existing
    `data/root` directory (i.e. true fresh-checkout conditions, matching
    CI), `run` succeeds (no ENOENT) and `verify` reports `good`.

## Evaluation Criteria
- **Good:** After the fix, deleting the local
  `e2e/symlink-inside-root-followed/data/root` directory and then running
  `./run && ./verify` (or `./test`) for this scenario from a clean state
  succeeds — no `ENOENT`, and `verify` prints `[GOOD] symlink physically
  under root was followed to its outside-root target`.
- **Good:** The next CI run of the `ubuntu e2e` job passes the
  `symlink-inside-root-followed` step (verifiable by checking CI logs
  after the fix is pushed).
- **Ambiguous:** N/A — this scenario is binary pass/fail, no partial states.
- **Bad:** Any recurrence of `ENOENT` on the `symlink()` call, or a
  passing local run that still fails in CI (would indicate the fix didn't
  address the real fresh-checkout gap, e.g. if some other leftover local
  state were masking a second issue).
